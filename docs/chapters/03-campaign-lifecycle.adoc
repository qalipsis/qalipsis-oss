= Campaign Lifecycle

A _campaign_ is a single execution of one or more scenarios.
This chapter covers the full lifecycle from creation through the REST API to completion, including the distributed state machine that orchestrates execution across multiple factory nodes.

== REST API Endpoints

All campaign endpoints are served by `CampaignController` in the head module.

[cols="1,3,3",options="header"]
|===
| Method | Path | Description

| POST
| `/campaigns`
| Start a new campaign with the given configuration

| POST
| `/campaigns/validate`
| Validate a campaign configuration without starting it

| GET
| `/campaigns`
| Search campaigns with filtering, sorting, and pagination

| GET
| `/campaigns/{campaignKeys}`
| Retrieve execution reports for one or more campaigns (comma-separated keys)

| GET
| `/campaigns/{campaignKey}/configuration`
| Retrieve the original configuration of a campaign

| POST
| `/campaigns/{campaignKey}/abort`
| Abort a running campaign (soft or hard)

| POST
| `/campaigns/{campaignKey}/replay`
| Replay a previously executed campaign with the same configuration

| POST
| `/campaigns/schedule`
| Schedule a campaign for future or recurring execution

| PUT
| `/campaigns/schedule/{campaignKey}`
| Update a scheduled campaign
|===

== Campaign State Machine

The campaign execution is driven by a state machine.
Each state processes feedbacks from factories and transitions to the next state by publishing new directives.

=== State Interface

Every campaign execution state implements `CampaignExecutionState`:

[cols="2,4",options="header"]
|===
| Method | Purpose

| `init()`
| Initialize the state and return directives to publish

| `process(feedback)`
| Process a feedback message and return the next state (or self)

| `abort(abortConfiguration)`
| Handle an abort request and return the aborting state

| `isCompleted`
| Whether this is a terminal state
|===

=== State Transitions Detail

==== FactoryAssignmentState

Publishes `FactoryAssignmentDirective` to each factory's unicast channel with its DAG assignments.

* **Waits for**: `FactoryAssignmentFeedback` (COMPLETED) from every assigned factory
* **Transitions to**: `MinionsAssignmentState` when all factories confirm
* **On failure**: Transitions to `FailureState`

==== MinionsAssignmentState

Publishes `MinionsDeclarationDirective` declaring the total minion count per scenario per factory.

* **Waits for**: `MinionsAssignmentFeedback` (COMPLETED) from every factory-scenario pair
* **Transitions to**: `MinionsScheduleRampUpState` when all assignments confirmed
* **On failure**: Transitions to `FailureState`

==== MinionsScheduleRampUpState (Warmup)

Publishes `MinionsRampUpPreparationDirective` for each scenario.
Each factory calculates its ramp-up schedule based on the execution profile.

* **Waits for**: `MinionsRampUpPreparationFeedback` (COMPLETED) from all factories
* **Transitions to**: `RunningState`
* **On failure**: Transitions to `FailureState`

==== RunningState

The main execution state.
Processes ongoing feedbacks:

* `MinionsStartFeedback` -- Records minion start (no transition)
* `CompleteMinionFeedback` -- Individual minion completed; triggers `MinionsShutdownDirective` for that minion
* `EndOfCampaignScenarioFeedback` -- All minions in a scenario finished; sends `CampaignScenarioShutdownDirective`
* `CampaignScenarioShutdownFeedback` -- Scenario cleanup complete; when all scenarios done, transitions to `CompletionState`
* `CampaignTimeoutFeedback` -- Soft timeout generates warnings; hard timeout triggers abort
* `FailedCampaignFeedback` -- Transitions to `FailureState`

==== CompletionState

Publishes `CampaignShutdownDirective` to all factories.

* **Waits for**: `CampaignShutdownFeedback` (COMPLETED) from all factories
* **Transitions to**: `DisabledState` with `ExecutionStatus.SUCCESSFUL`

==== FailureState

Sends shutdown directives and cleans up resources.

* **Transitions to**: `DisabledState` with `ExecutionStatus.FAILED`

==== AbortingState

Sends `CampaignAbortDirective` to all factories for the specified scenarios.

* **Waits for**: `CampaignAbortFeedback` from all healthy factories
* **Transitions to**: `DisabledState` with `ExecutionStatus.ABORTED`

== Timeout Handling

Campaigns support both soft and hard timeouts:

[cols="2,2,3",options="header"]
|===
| Timeout | Effect | Behavior

| Soft timeout
| Warning
| `CampaignTimeoutFeedback(hard=false)` is generated. Campaign continues but a warning event is logged. Useful for alerting that a campaign is running longer than expected.

| Hard timeout
| Abort
| `CampaignTimeoutFeedback(hard=true)` is generated. Triggers transition to `AbortingState`, sending `CampaignAbortDirective` to all factories. Campaign terminates with `ABORTED` status.

| No timeout
| None
| Set `hardTimeoutSec = -1` and `softTimeoutSec = -1`. Campaign runs until all minions complete or an error occurs.
|===

== Distributed State Persistence

In Redis-based deployments, campaign state is persisted to Redis for failover:

* `CampaignRedisOperations` manages state storage
* `CampaignRedisState` enum maps to state classes: `FACTORY_DAGS_ASSIGNMENT_STATE`, `MINIONS_ASSIGNMENT_STATE`, `WARMUP_STATE`, `MINIONS_STARTUP_STATE`, `RUNNING_STATE`, `COMPLETION_STATE`, `FAILURE_STATE`, `ABORTING_STATE`
* On head recovery, `RedisCampaignExecutor` rebuilds state machines from persisted Redis data
