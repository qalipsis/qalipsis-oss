=== Channel Types

[cols="3,2,3",options="header"]
|===
| Channel Pattern | Direction | Purpose

| `handshake-request`
| Factory -> Head
| Factory registration requests

| `factory-{nodeId}-replyTo`
| Head -> Factory
| Handshake response (factory-specific)

| `heartbeat`
| Factory -> Head
| Periodic health signals

| `campaign-{key}-broadcast`
| Head -> All Factories
| Campaign-wide directives (e.g., shutdown)

| `factory-{nodeId}-unicast`
| Head -> Specific Factory
| Factory-specific directives (e.g., assignment)

| `campaign-{key}-feedback`
| Factory -> Head
| Campaign feedback messages
|===

=== RedisHeadChannel

Published methods:

[cols="3,4",options="header"]
|===
| Method | Description

| `publishDirective(directive)`
| Send a directive to the appropriate channel (broadcast or unicast)

| `publishHandshakeResponse(channel, response)`
| Send handshake response to factory's replyTo channel

| `subscribeHandshakeRequest(channels)`
| Start listening for handshake requests

| `subscribeFeedback(channels)`
| Start listening for campaign feedbacks
|===

=== RedisFactoryChannel

Published methods:

[cols="3,4",options="header"]
|===
| Method | Description

| `publishFeedback(feedback)`
| Send feedback to the campaign's feedback channel

| `publishHeartbeat(channel, heartbeat)`
| Send periodic heartbeat to the heartbeat channel

| `subscribeHandshakeResponse(channels)`
| Listen for handshake responses

| `subscribeDirective(channels)`
| Listen for directives on broadcast and unicast channels
|===

=== Message Serialization

All messages are serialized using Kotlin Serialization with polymorphic type support:

* `DirectiveRegistry.prepareBeforeSend()` transforms directives before sending (e.g., storing `SingleUseDirective` payloads in Redis and replacing with references)
* `DirectiveRegistry.prepareAfterReceived()` resolves references back to full directives on the receiving side
* This reduces Redis message size for large payloads like minion assignment tables

=== Tenant Isolation Enforcement

1. **REST Layer**: The `@Tenant` annotation on controller parameters injects the tenant reference from the request context (typically extracted from the JWT token or `X-Tenant` header)
2. **Service Layer**: All service methods receive the tenant parameter and pass it to repository queries
3. **Database Layer**: All entities include a `tenant_id` foreign key; all queries are filtered by tenant
4. **Communication Layer**: Directives, feedbacks, and heartbeats all carry the `tenant` field for routing

=== Permission-Based Access Control

Endpoints are secured with the `@Secured` annotation specifying required permissions:

[source,kotlin]
----
@Post("/campaigns")
@Secured(Permissions.WRITE_CAMPAIGN)
suspend fun start(@Tenant tenant: String, @Body configuration: CampaignConfiguration)
----

.Zone Data Model
[cols="2,3",options="header"]
|===
| Field | Description

| `key`
| Short code (e.g., `US`, `EU`)

| `title`
| Display name (e.g., `United States East`)

| `description`
| Details (e.g., `us-east-1, us-east-2`)

| `imagePath`
| URL to a logo or flag for the UI

| `enabled`
| Whether the zone can be used in campaigns
|===

Factories register with a zone during handshake.
Campaign configurations specify how many minions should run in each zone.

== External Plugin Support

QALIPSIS supports 17+ plugins for protocol-specific steps:

[cols="2,2,3",options="header"]
|===
| Category | Plugins | Description

| HTTP/Network
| Netty
| HTTP/1.1, HTTP/2, TCP, UDP, MQTT

| Databases
| Cassandra, Elasticsearch, InfluxDB, MongoDB, R2DBC-Jasync, TimescaleDB
| Read/write operations for various databases

| Messaging
| Kafka, RabbitMQ, JMS, Jakarta EE
| Produce/consume messages

| Metrics
| Graphite
| Export metrics via Pickle/Plaintext

| Data Formats
| Jackson
| JSON, XML, CSV parsing and conversion

| Notifications
| Slack, Mail
| Send notifications during tests

| Caching
| Redis-Lettuce
| Redis read/write operations
|===

Each plugin provides specialized `Step` implementations and `StepConverter` annotations, which are discovered at compile time via KAPT annotation processors.
