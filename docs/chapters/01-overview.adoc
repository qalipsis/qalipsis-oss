= Architecture Overview

QALIPSIS is an enterprise-grade, open-source load, performance, and end-to-end testing platform for distributed systems.
Built in Kotlin on the JVM, it simulates thousands to millions of virtual actors (called _minions_) executing user-defined _scenarios_ against a system under test.

.Module Descriptions
[cols="2,4",options="header"]
|===
| Module | Purpose

| `api-common`
| Core interfaces and data structures: steps, meters, events, scenarios, execution context

| `api-dsl`
| Kotlin DSL for writing load test scenarios with fluent step chaining

| `api-processors`
| Compile-time annotation processors (KAPT) for generating step converters

| `api-dev`
| Development platform configuration for scenario projects

| `shared`
| Shared protocol definitions: directives, feedbacks, handshake, heartbeat, campaign structures

| `cluster`
| Multi-tenancy support, security, and permission management

| `head`
| Head microservice: REST API, campaign orchestration, factory management, reporting

| `factory`
| Factory microservice: directive processing, minion orchestration, step execution, metrics

| `runtime`
| Bootstrap and execution environment; supports standalone, head-only, and factory-only modes

| `dev-platform` / `plugin-platform`
| BOM modules defining dependency constraints for scenario and plugin developers

| `webapp`
| Vue.js/Nuxt web UI for campaign management and result visualization
|===

== Technology Stack

[cols="2,3",options="header"]
|===
| Technology | Purpose

| Kotlin 1.9.25
| Primary language; coroutines for async execution

| Micronaut 3.9.7
| Microservice framework (DI, HTTP server, validation)

| Redis (Lettuce)
| Inter-node messaging via Pub/Sub (directives, feedbacks, heartbeats)

| PostgreSQL
| Persistent storage for campaigns, scenarios, reports, factories

| Hazelcast 4.2.8
| Distributed caching and coordination

| Kotlin Serialization
| Polymorphic message serialization for protocol messages

| Kotlin Coroutines 1.8.1
| Concurrent minion execution and async I/O

| Jackson 2.18.4
| JSON serialization for REST API

| JUnit 5 / MockK / AssertK
| Testing framework
|===

== Communication Topology

[plantuml,communication-topology,svg]
----
@startuml

title Redis Communication Topology

participant "Head" as head
collections "Redis Pub/Sub" as redis
participant "Factory A" as fa
participant "Factory B" as fb

== Factory Registration ==

fa -> redis : HandshakeRequest\n(handshake-request channel)
redis -> head : HandshakeRequest
head -> redis : HandshakeResponse\n(factory-specific replyTo channel)
redis -> fa : HandshakeResponse\n(nodeId, unicastChannel,\nheartbeatChannel, heartbeatPeriod)

fb -> redis : HandshakeRequest
redis -> head : HandshakeRequest
head -> redis : HandshakeResponse
redis -> fb : HandshakeResponse

== Health Monitoring ==

fa -> redis : Heartbeat\n(heartbeat channel)
redis -> head
fb -> redis : Heartbeat\n(heartbeat channel)
redis -> head

== Campaign Execution ==

head -> redis : Directives\n(broadcast or unicast channel)
redis -> fa : FactoryAssignment,\nMinionsDeclaration, ...
redis -> fb : FactoryAssignment,\nMinionsDeclaration, ...

fa -> redis : Feedbacks\n(feedback channel)
redis -> head : FactoryAssignment,\nMinionsAssignment, etc.
fb -> redis : Feedbacks\n(feedback channel)
redis -> head

@enduml
