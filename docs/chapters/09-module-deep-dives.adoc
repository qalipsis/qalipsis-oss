=== Campaign Execution Context

The `CampaignExecutionContext` is a singleton injected into all campaign states:

[cols="2,3",options="header"]
|===
| Dependency | Purpose

| `campaignService`
| Persist campaign state transitions

| `factoryService`
| Query and manage factory assignments

| `campaignReportStateKeeper`
| Track report generation state

| `headChannel`
| Publish directives to factories

| `assignmentResolver`
| Resolve factory-scenario DAG assignments

| `reportPublishers`
| Publish reports after completion

| `campaignAutoStarter`
| Trigger auto-start behavior (if configured)

| `campaignHooks`
| Extension hooks for campaign lifecycle events

| `delayedFeedbackManager`
| Handle feedbacks that arrive before state is ready
|===

|===

| Collects and publishes metrics during execution

| `EventsLogger`
| Logs events during minion and step lifecycle
|===

=== Directive Listener Pattern

All directive listeners implement the `DirectiveListener<D>` interface:

[source,kotlin]
----
interface DirectiveListener<D : Directive> : Ordered {
    fun accept(directive: Directive): Boolean
    suspend fun notify(directive: D)
}
----

Listeners are invoked in priority order (`Ordered` interface).
Each listener:

1. Checks if it should handle the directive via `accept()`
2. Publishes a `Feedback(IN_PROGRESS)` before processing
3. Executes the directive logic
4. Publishes `Feedback(COMPLETED)` or `Feedback(FAILED, errorMessage)`

== Runtime Module

The runtime module provides the bootstrap and execution environment.

=== Deployment Roles

[cols="2,4",options="header"]
|===
| Role | Description

| `AUTO`
| Auto-detect based on classpath and configuration

| `STANDALONE`
| Both head and factory in a single JVM

| `HEAD`
| Only head components loaded

| `FACTORY`
| Only factory components loaded
|===

=== Bootstrap Flow

[plantuml,bootstrap-flow,svg]
----
@startuml

title Runtime Bootstrap Flow

start

:Qalipsis.main(args);

:Parse command-line arguments\n(deployment role, autostart, etc.);

if (Role?) then (STANDALONE)
    :Load Head + Factory components;
    :Initialize in-memory channels;
elseif (Role?) then (HEAD)
    :Load Head components only;
    :Initialize Redis channels;
elseif (Role?) then (FACTORY)
    :Load Factory components only;
    :Initialize Redis channels;
    :Initiate handshake with Head;
else (AUTO)
    :Detect from classpath;
endif

:Start Micronaut application context;

if (--autostart?) then (yes)
    :CampaignAutoStarter\nruns campaign immediately;
    :Wait for completion;
    :Exit with status code;
else (no)
    :Server ready\n(waiting for REST/Redis commands);
endif

stop

@enduml
----

=== In-Memory vs. Redis Execution

[cols="2,3,3",options="header"]
|===
| Concern | In-Memory (Standalone) | Redis (Cluster)

| Directives
| Direct method calls
| Redis Pub/Sub channels

| Feedbacks
| Direct method calls
| Redis Pub/Sub channels

| State persistence
| JVM heap
| Redis keys

| Handshake
| Implicit (same JVM)
| Redis handshake protocol

| Heartbeat
| Not needed
| Periodic Redis messages
|===

== API Common Module

Defines the core interfaces used by scenario developers and plugin authors.

=== Key Interfaces

[cols="2,4",options="header"]
|===
| Interface | Purpose

| `Step<I, O>`
| Atomic unit of work with lifecycle hooks (init, start, execute, stop, complete, destroy)

| `StepContext<IN, OUT>`
| Execution context carrying data between steps

| `Meter<SELF>`
| Base meter interface with snapshot and summary methods

| `Counter`, `Timer`, `Gauge`, `DistributionSummary`
| Concrete meter types

| `EventsLogger`
| Interface for logging events during execution

| `Scenario`
| Runtime representation of a scenario with DAGs and steps

| `DirectedAcyclicGraph`
| Container for a chain of steps

| `Minion`
| Virtual actor executing step chains
|===
